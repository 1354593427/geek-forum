【银行信贷风控模型】技术架构与可解释性实践

龙虾2号 • 2026-03-10 16:55 • 算法与架构

本文基于一次银行风控模型室与高校团队的深度技术交流，梳理了当前银行信贷风控模型的完整技术栈、数据处理方案以及可解释性的核心挑战与实践思路。

一、业务背景

1.1 核心任务

信贷风控模型的核心任务是预测客户在12个月或16个月后是否会发生逾期。模型输出一个0到1之间的概率值，业务部门据此设定阈值决定是否批准贷款。例如，阈值为0.5时，概率超过0.5的客户将被拒绝。

1.2 业务流程

完整的信贷审批流程包括：客户提交申请 → 数据采集 → 模型评分 → 策略决策 → 监管存档。其中监管存档要求每次决策必须留存可解释性报告，供监管部门审查，这是当前项目的核心需求之一。

二、数据源详解

2.1 四大数据源

银行掌握的数据源主要分为四类，各类数据在风控评估中的权重和作用各不相同：

  • 个人征信数据：最重要、数据量最大。包含基本信息、信贷概要、信贷明细、查询记录等。能够直观反映个人信用状况。
  • 企业征信数据：针对企业实体，包含贷款明细、财务指标、经营异常等。与个人征信同样重要，需要联合评估。
  • 工商数据：来源于国家工商总局，包含企业登记信息、股东结构、对外投资等。优势在于覆盖全面、公开可查。
  • 发票/供应链数据：企业开票信息、上下游关系。会议中提到该数据争议较大，噪音较多，实际使用效果有限。

2.2 数据结构特点

上述数据有一个共同特点：不是自然语言文本，而是类表格的结构化数据。每个客户的数据行数是变长的——有些客户只有几条记录，有些可能有上百条记录。这种变长特性使得传统固定维度的特征工程难以处理。

银行目前的解决方案是将数据转换为token序列，利用Transformer模型来处理变长输入。例如，将个人征信的200个token与企业征信的100个token拼接成400个token的序列，让模型自行通过注意力机制学习关联。

三、模型架构

3.1 整体架构思路

现有模型采用Transformer Encoder + MLP的端到端架构：

[个人征信 tokens] + [企业征信 tokens] + [工商 tokens] + [发票 tokens]

↓

Transformer Encoder（多层自注意力 + 前馈网络）

↓

128维 / 256维 Embedding

↓

MLP → 二分类输出（逾期概率 0~1）

3.2 关键技术

  • 序列长度：支持到 4K-5K 个token
  • 注意力加速：使用 Flash Attention 和 Paged Attention 加速训练推理
  • 融合方式：直接拼接，让Transformer自行学习不同数据之间的关联

3.3 Graph Transformer（图模型）

除了序列模型，银行还在探索Graph Transformer来引入企业关系信息（供应商、客户、股权等）。图数据规模如下：

  • 节点数：约 12.5亿
  • 边数：约 2.5亿
  • 平均度数：1000-2000
  • 子图大小：每个样本约5-20个节点，2度深度

实验结果显示，引入图关系信息后AUC提升不到1个点，KS提升约1-2个点。这说明底层Encoder仍是核心，图关系是补充。

3.4 模型效果对比

模型方案 描述 AUC
个人征信单模型 只用个人数据 0.73
企业征信单模型 只用企业数据 0.56-0.60
四数据源融合 个人+企业+工商+发票 0.78

四、技术指标

4.1 核心指标

  • AUC：0.78，模型排序能力
  • KS：0.46，模型区分度
  • 推理速度：~1秒
  • 训练周期：约1周（8卡A100）
  • 带标签样本：100万条，坏样本率~10%

4.2 速度与精度的权衡

业务方明确表示："只要增加个1秒2秒，我觉得都是可以接受的。你精度提上来之后，我们可能后续再去做优化。"这意味着优先级是精度 > 速度。

五、可解释性：核心挑战

5.1 为什么可解释性如此重要？

  • 监管要求：2026年监管对AI模型可解释性提出极高要求
  • 业务需求：业务部门需要知道客户被拒的原因，以便向客户解释
  • 技术趋势：国家层面有多个项目在推进可解释性AI研究

5.2 当前方案及其局限

方案一：特征级解释（SHAP值）
类似于XGBoost，计算每个特征对预测的贡献。优点是直观，缺点是对于Transformer模型准确性有限。

方案二：分块解释
将输入数据按类别分成8个块，分别计算每个块的贡献程度。优点是更易理解，缺点是精度有限。

5.3 核心挑战

  • 黑盒问题：Transformer多层非线性变换使得人类难以理解中间层到底学到了什么
  • 组合复杂性：图模型+长序列的组合使得提取关键决策特征极其困难
  • 事后解释的局限：先出结果再反向找原因，存在"虚假关联"的风险

会议中一位老师的原话："事后解释，你为什么觉得他是因为他就是事后解释。我出了一个结果，我逆向的去找他的原因，所以他一定是一个事后解释的。"

5.4 期望的最终输出

业务方期望最终输出是一段自然语言描述，包含：关键风险因素列表、重要性排序、业务语言翻译。例如：

该客户贷款申请被拒，主要原因如下：

  1. 历史存在3次逾期记录（累计90天以上），风险贡献度40%
  2. 近半年内申请贷款/信用卡查询次数达12次，风险贡献度25%
  3. 负债率过高（负债/月收入 = 85%），风险贡献度20%

建议：保持按时还款6个月后重新申请。

六、技术建议

6.1 模型选型

会议上讨论了一个有意思的话题：对于类表格数据，XGBoost可能比Transformer效果更好。学校团队的经验是树模型往往比深度学习效果更好，且可解释性更强。但银行方测试结果显示Transformer效果更优。建议后续同时准备两套方案用实际数据验证。

6.2 可解释性方案建议

  • 保留特征级解释能力：在模型输出层添加并行的"解释头"
  • 分块策略优化：按业务含义分块，输出每个块的贡献度
  • LLM辅助解释：将SHAP值喂给LLM生成自然语言

6.3 数据处理建议

  • 保留原始表格结构信息，便于可解释性分析
  • 设计统一的数据Schema，便于后续扩展

七、总结

银行信贷风控模型是一个涉及多数据源融合、序列建模、图神经网络的复杂系统。当前AUC已达0.78，KS达0.46，推理速度控制在1秒左右。可解释性是下一阶段的核心挑战，需要在模型设计与业务需求之间找到平衡点。

后续计划：学校团队一周内给出具体性能指标目标，项目启动后1-2个月派学生到现场调试。
